## 🎯 프로젝트 개요

### 배경 & 동기

반도체 팹의 OHT 시스템은 100대 이상의 로봇이 동시에 실시간으로 움직이는 분산 시스템이다. 교착(deadlock)이 발생하면 생산 라인 전체가 멈추기 때문에 안정성이 무엇보다 중요하다.

**목표**:
1. **알고리즘**: 100대 로봇이 안정적으로 운행하는 경로탐색 알고리즘 설계
2. **통합 구현**: 웹 기반 맵 에디터 + 3D 시뮬레이터 + 알고리즘 비교 페이지
3. **포트폴리오화**: 개발 과정과 기술 의사결정을 명확히 기록

### 성과 요약

| 지표 | 결과 |
|------|------|
| **교착률** | 관찰된 테스트에서 거의 0% (Standard A*는 빈번한 교착) |
| **동시 로봇 수** | 100대까지 안정 운행 확인 |
| **알고리즘 종류** | 7가지 (Standard A*, Dijkstra, Greedy BFS, Stochastic A*, CBS Full, Priority A*, CBS-Lite/WHCA*) |
| **노드 종류** | 6가지 (Normal, Deposition, Exposure, Etching, Cleaning, Depot) |
| **플랫폼** | 웹 에디터 (React) + 3D 시뮬레이터 (Unity) |
| **코드 규모** | TypeScript ~3,500 LOC + C# ~2,000 LOC (개략치) |

## 🧠 기술 결정사항

### 1️⃣ 경로탐색 알고리즘: 시도와 실패에서 도출한 하이브리드

100대 로봇이 동시 운행하는 환경에서 "어떤 경로탐색이 안전한가"를 6단계에 걸쳐 직접 검증했다. 각 단계는 단순한 비교가 아니라, **이전 단계에서 관찰한 실패 모드가 다음 시도의 근거**가 되는 순서로 진행됐다.

#### Round 1 — Standard A* (표준 A*): 빠르지만 충돌을 모른다
- **시도**: 휴리스틱 + 실제 비용 기반의 표준 A*로 각 로봇이 독립적으로 최단 경로 계산 (`standardAStar` / `algorithms.ts`)
- **관찰된 실패 모드**:
  - 두 로봇이 동일 노드에 동시 도착 → 데드락
  - 단방향 그리드 토폴로지 도입 전에는 양방향 간선에서 정면충돌까지 발생
  - 모두가 같은 최단 경로로 몰리면서 핫스팟에 병목 형성
- **교훈**: 시공간 정보 없이 "개인 최적 경로"만 계산하는 단발 탐색은 다중 에이전트 환경에서 구조적으로 불충분하다

#### Round 2 — Dijkstra / Greedy BFS: 휴리스틱만 바꿔도 충돌은 그대로다
- **시도**: 휴리스틱을 끈 Dijkstra(`dijkstra`)와 휴리스틱만 쓰는 Greedy BFS(`greedyBFS`)로 비교
- **관찰된 실패 모드**:
  - Dijkstra: A*와 동일한 최단 경로를 더 느리게 찾을 뿐, 협력은 여전히 0
  - Greedy BFS: 목표 방향만 따라가다 혼잡 구역에서 루프/우회 실패, 최단성도 미보장
  - 두 알고리즘 모두 Round 1과 같은 데드락 양상을 그대로 재현
- **교훈**: 필요한 것은 "더 똑똑한 단발 경로"가 아니라 **여러 로봇 사이의 조정 메커니즘**이다. 휴리스틱 튜닝으로는 해결 불가능한 클래스의 문제

#### Round 3 — Stochastic A* (확률적 A*): 노이즈로 우회를 강제
- **시도**: 간선 비용에 ±30% 노이즈를 곱해(`stochasticAStar`, `noise = 0.3`) 로봇마다 약간씩 다른 경로 선택을 유도
- **관찰된 실패 모드**:
  - 일시적 분산 효과는 있지만 통계적 우연에 의존 — 100대 환경에선 여전히 충돌 발생
  - 같은 맵·같은 시드여도 비결정적이라 실패 재현/디버깅이 어려움
- **교훈**: 임의성은 "운으로 회피"하는 임시방편. 명시적 조정 신호가 필요하다

#### Round 4 — Priority A* (혼잡 가중 A*): 공간 혼잡도를 비용에 반영
- **시도**: 도착 노드의 혼잡도를 간선 비용에 반영 — `cost = weight × (1 + congestion × 2.5)` (`priorityAStar`)
- **관찰된 실패 모드**:
  - 분산 효과는 크게 개선 — 핫스팟이 명시적으로 비싸짐
  - 그러나 혼잡도는 **현재 시점의 공간 점유**만 반영 → 같은 시점·같은 노드에 도달하는 **시공간 충돌**은 여전히 발생 가능
- **교훈**: 공간 차원의 혼잡 회피만으로는 시간 차원의 충돌을 막을 수 없다. 시공간을 모두 보는 메커니즘이 필요

#### Round 5 — CBS Full 시도 → 실시간 포기
- **시도**: Conflict-Based Search 전체 구현. 고수준 충돌 트리(Constraint Tree) + 저수준 A* 재탐색의 2계층 구조를 Unity C#으로 완성
- **관찰된 실패 모드**:
  - 2~4대 환경에서는 완벽한 충돌 회피와 최적 경로
  - 로봇 수가 늘수록 Constraint Tree가 지수적으로 분기 → 실시간 프레임 예산 초과
- **교훈**: CBS Full은 오프라인/소규모 정적 환경 전용. 실시간 100대에서는 **근사(Lite)** 가 필요하다

#### Round 6 — 채택: Priority A* + WHCA*(CBS-Lite) 하이브리드 ✅
- **결합 방식**:
  - 기본 경로 = **Priority A*** — 혼잡도 맵(0~1)을 엣지 비용에 ×2.5 페널티로 반영 → 자연 분산
  - 보조 충돌 회피 = **WHCA* (Windowed Hierarchical Cooperative A*)** — 가까운 미래 **12스텝**에 한해 (NodeId, Timestep) 시공간 예약, 후순위 에이전트는 예약 노드에 비용 ×8 페널티로 우회
  - 진동 억제 = **PathCommitmentSteps = 5** — 매 틱 재계산이 아닌 5스텝 이동 후에만 재탐색
  - 자료구조 최적화 = OpenList를 선형 탐색에서 **BinaryHeap<Node>** 로 교체 (O(log V))
- **왜 이 조합인가**: 공간(혼잡) + 시간(예약) 두 차원을 분리해 각각 가장 적합한 도구로 처리. Priority A*가 사전 분산을 잘 하므로 WHCA* 페널티가 실제로 발동하는 빈도가 낮다 → 비용 ↓
- **결과(자체 관찰)**: 100대 / 12×8 맵 / 장시간 운행에서 교착 거의 0%

#### Round 7 — 데드락 에스컬레이션 L1 ~ L4
하이브리드가 막아도 살아남는 잔여 교착을 단계적으로 해소.

| 단계 | 트리거 | 동작 |
|------|--------|------|
| L1 | 매 프레임 | Wait-for 그래프 DFS 사이클 감지 → 최저 우선순위 에이전트 ForceReroute (관찰상 약 70% 해소) |
| L2 | 5초 정지 | 교차로 정지 에이전트를 Blocked 처리, 매 프레임 대안 경로 재계산 |
| L3.5 | 5초+ 완전 포위 | 인접 4방향 모두 정지 시 차단 로봇들에게 TryPhysicalYield 브로드캐스트 |
| L3 | 12초 정지 | 인접 빈 노드로 1칸 강제 이동(물리적 공간 확보) 후 경로 재탐색 |
| L4 | 최후 수단 | AbandonJob → Idle 복귀, Dispatcher가 재배정 |

> 단계별 상세 로직은 웹 `/algorithm` 페이지 § 7 참고.

#### 회고
- **알고리즘 자체보다 "실패 모드를 정확히 관찰하는 것"** 이 다음 시도의 근거가 됐다 (Round N의 교착 양상 → Round N+1의 설계 결정)
- **단방향 그리드**라는 토폴로지 설계가 "정면충돌"이라는 한 클래스의 실패를 구조적으로 제거 → 남은 문제(동시 점유)에 집중할 수 있게 됨
- 결론은 단일 "완벽한 알고리즘"이 아니라 **공간/시간을 분리한 하이브리드** — 한 알고리즘의 약점을 다른 알고리즘이 메우는 구조가 실용적이었다

> **주의**: 위 비교는 자체 테스트(대형 12×8 맵, 100대 로봇)에서의 정성적 관찰에 기반한다. 정량적 CPU 효율 수치나 처리율 절대값은 통계적 신뢰구간을 갖춘 벤치마크가 아직 없다.

#### 의외의 발견 — 시스템이 알고리즘을 압도한다

7종 길찾기 알고리즘을 단계적으로 비교했지만, **자동 디스패칭**(가장 가까운 가용 로봇 ↔ 가장 가까운 작업 자동 배정 + 모든 로봇이 함께 보는 혼잡도 맵을 공유)을 적용한 순간, 알고리즘별 처리량/혼잡도 차이가 **노이즈 수준**으로 줄어들었다. Standard A*조차 자동 디스패칭 위에서는 Priority A* + WHCA* 하이브리드와 구분이 어려울 정도였다.

원인은 다음과 같이 정리된다.
- **목표 자동 분산**: 디스패처가 작업을 공간적으로 흩뿌리니, 로봇들이 같은 최단 경로로 몰리는 구조 자체가 사라진다.
- **공유된 혼잡 정보**: 모든 알고리즘이 같은 혼잡도 신호를 비용에 반영하므로, 휴리스틱 차이가 만들어내던 격차가 평탄화된다.
- **"어디로 가느냐"보다 "누가 어디로 배정되느냐"가 지배적**: 경로탐색의 최적성이 아니라 배정의 균형이 시스템 처리량의 상한을 결정한다.

개별 컴포넌트보다 전체 흐름의 설계가 성능 상한을 결정한다는 것은 다중 에이전트 시스템에서 자주 관찰되는 패턴이다.

> **결론적 교훈**: 본 프로젝트의 진짜 기여는 "더 나은 길찾기 알고리즘 선택"이 아니라, **경로탐색 + 자동 디스패칭 + 데드락 에스컬레이션을 결합한 시스템 설계** 그 자체다. 어떤 단일 알고리즘을 골랐는지가 아니라, 세 층이 같은 혼잡도 신호 위에서 협력하도록 묶은 구조가 100대 운행을 가능하게 했다.

> **단, 한계**: 이 관찰은 **자동 디스패칭 ON, 100대 / 12×8 맵 조건**에서의 정성적 결과다. 디스패칭을 끈 수동 배정, 혹은 극단적 혼잡(200대+, 좁은 통로, 병목 토폴로지)에서는 알고리즘 간 차이가 다시 드러날 가능성이 충분하다. 일반화하려면 추가 벤치마크가 필요하다.

### 2️⃣ 맵 설계: 단방향 그리드 (Unidirectional Grid Topology)

**원리**: 행/열 패리티를 이용한 일방향 이동.

```text
행 패리티:          열 패리티:
짝수 행 → 우측      짝수 열 → 하향
홀수 행 ← 좌측      홀수 열 ← 상향

결과: 2-cycle 구조적 불가능 → 정면충돌 0% (구조적 보장)
```

**설계 제약**:
- 특수 노드(증착·노광·식각·세정·차고지)는 (짝수행, 짝수열) 부분격자에만 배치
- 이를 통해 특수 노드 간 비인접성 + 균등 분포 보장

**프리셋 (모두 단방향, 코드의 `buildOneWayGrid` 호출 기준)**:

| 맵 | 크기 (cols × rows) | 노드 수 | 차고지 | 용도 |
|----|--------------------|---------|--------|------|
| 소형 | 6 × 4 | 24 | 2 | 학습용 |
| 중형 | 8 × 6 | 48 | 4 | 기본/테스트 |
| 대형 | 12 × 8 | 96 | 6 | 포트폴리오 데모 |
| 초대형 | 20 × 16 | 320 | 8 | 스트레스 테스트 |

### 3️⃣ 웹 ↔ Unity 포팅: TypeScript ↔ C# 동일성

**과제**: 같은 알고리즘을 두 언어/플랫폼에서 구현하고 동일 결과를 보장.

**해결**:
1. **Web (TypeScript)**: 인터랙티브 데모 / 알고리즘 비교 페이지
2. **Unity (C#)**: 3D 시각화 / 성능 검증 환경
3. **검증**: 같은 맵 + 같은 시드로 동일 경로가 나오는지 비교

**구현 시그니처 비교**:

```typescript
// TypeScript (웹)
export function priorityAStar(from: RailNode, to: RailNode, ...): Path
```

```csharp
// C# (Unity)
public static List<int> PriorityAstar(MapNode from, MapNode to, Dictionary<string, float> congestion)
```

### 웹 에디터 (React + Konva.js)

**기능**:

### 알고리즘 검증 페이지 (`/algorithm`)

### 관찰 결과 (정성적)

| 알고리즘 | 교착 발생 | 비고 |
|----------|-----------|------|
| Standard A* | 빈번하게 관찰됨 | 단방향 그리드여도 시공간 충돌 처리가 없어 잠금 발생 |

## ⚠️ 추가로 고려해야 할 점 (현재 한계)

이 프로젝트가 **포트폴리오 데모 단계**임을 솔직하게 인정한다. 실제 팹 환경에 적용하려면 아래 항목들이 필요하다.

### 1) 물리/제어 모델 부재
- 가감속(acceleration / deceleration) 모델 없음 — 현재는 등속 가정
- 비상정지(E-stop), 우선정지(priority stop), 회피 트리거 같은 안전 로직 미구현
- 실제 팹 레이아웃(곡선 레일, 분기, 교차 게이트, 층간 이동) 호환성 미검증

### 2) 알고리즘 한계
- 100대까지는 안정 관찰 — **그 이상(>100대) 규모는 검증되지 않음**
- WHCA* 12스텝 윈도우는 초대형 맵(20×16, 320 노드)에서 비용이 어떻게 스케일링되는지 측정 필요
- 동적 장애물, 로봇 고장, 우선순위 충돌 같은 시나리오 미고려

### 3) 검증 한계
- 테스트는 단일 워크스테이션에서 수행 — 하드웨어 다양성/장기 안정성 미검증
- 자동화된 벤치마크 하네스 없음 → 통계적 신뢰구간(CI) 산출 불가
- 회귀 테스트 스위트 없음 → 알고리즘 변경 시 성능 변화를 자동 감지하지 못함

### 4) 향후 보완 방향
- 정량 벤치마크 자동화 (시드/맵/알고리즘 매트릭스 + 신뢰구간 산출)
- 가감속 + 안전 로직 모델링
- 실제 팹 레이아웃 XML 임포트 파이프라인
- >100대 규모 스트레스 테스트

## 📚 부록: 기술 참고

### A. WHCA* (CBS-Lite) 공간-시간 예약 알고리즘

```text
시간 축:  t0  t1  t2  t3  t4  t5 ... t11
         ┌──────────────────────────────┐ (12스텝 윈도우)
Node A   │███│   │   │▒▒▒│   │   ...   │
         └──────────────────────────────┘
                    ↑ 미래 충돌 감지
         (NodeId=A, Timestep=3) 예약됨 → 비용 ×8 페널티
```

### B. Priority A* 혼잡도 가중치

```text
혼잡도 = (주변 노드 중 점유된 비율)

간선 비용 = base_cost × (1.0 + congestion × 2.5)

예시:
- 혼잡도 0    → 비용 = base × 1.0
- 혼잡도 0.5  → 비용 = base × 2.25
- 혼잡도 1.0  → 비용 = base × 3.5
```

### C. 단방향 그리드 토폴로지 증명

**명제**: 인접한 두 노드 A, B 사이에 2-cycle (A→B→A)이 불가능하다.

**증명**:
1. A → B 이동이 가능 ⟹ 행 패리티 또는 열 패리티 규칙에 의해 한 방향이 결정됨
2. B → A 이동은 그 반대 방향
3. 같은 패리티 규칙에 의해 역방향 이동은 허용되지 않음
4. 따라서 2-cycle은 구조적으로 불가능 ∎
