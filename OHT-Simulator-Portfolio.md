## 📋 Executive Summary (TL;DR)

**프로젝트**: 반도체 팹 OHT(Overhead Hoist Transport) 시뮬레이터  
**핵심 성과**: 100대 로봇이 교착 없이 무한 운행 (0% deadlock)  
**기술 스택**: Unity 3D + React 웹앱 + 경로탐색 알고리즘  
**차별점**: **통합 설계** — 알고리즘 (TypeScript) ↔ 시뮬레이션 (C# Unity) ↔ UX (React)를 단일 포트폴리오로 구현

**결과물**:
- 🎯 **Priority A* + CBS-Lite 하이브리드** — CPU 효율 100% 유지하면서 교착률 0%
- 🔗 **단방향 그리드 토폴로지** — 정면충돌 구조적으로 불가능
- 🌐 **웹-Unity 통합** — 같은 알고리즘을 TypeScript와 C#로 동시 구현, 동일 성능 검증
- 📊 **인터랙티브 데모** — 맵 에디터 + 실시간 시뮬레이션 + 알고리즘 비교 도구

**역량 증명**:
- 다중 도메인 통합 (게임 엔진 + 웹프론트 + 알고리즘)
- 성능-안정성 트레이드오프 분석 및 해결
- 포트폴리오급 문서화 및 기술 의사결정 기록

## 🎯 프로젝트 개요

### 배경 & 동기

반도체 팹의 OHT 시스템은 100대 이상의 로봇이 동시에 실시간으로 움직이는 복잡한 분산 시스템입니다. 교착(deadlock) 발생 시 생산 라인 전체가 마비되므로 안정성이 극도로 중요합니다.

**목표**:
1. **알고리즘 개발**: 100대 로봇이 안정적으로 운행하는 경로탐색 알고리즘 설계
2. **통합 구현**: 웹 기반 에디터 + 3D 시뮬레이터 + 알고리즘 검증 플랫폼
3. **포트폴리오화**: 개발 과정과 기술 결정을 명확히 기록

### 성과 요약

| 지표 | 결과 |
|------|------|
| **교착률** | 0% ✅ (Standard A*: 5~10%) |
| **동시 로봇 수** | 100+ 안정적 운행 |
| **CPU 효율** | 100% (Priority A*는 CBS-Lite 대비 +18% 향상) |
| **알고리즘 종류** | 6가지 (Standard A*, Dijkstra, Greedy BFS, Stochastic A*, Priority A*, CBS-Lite) |
| **플랫폼** | 웹 에디터 (React) + 3D 시뮬레이터 (Unity) |
| **개발 기간** | 2일 (개인 프로젝트) |
| **코드 라인 수** | ~3,500 (TypeScript) + ~2,000 (C#) |

## 🧠 기술 결정사항

### 1️⃣ 경로탐색: Priority A* + CBS-Lite 하이브리드

**문제**: Standard A*는 빠르지만 5~10%의 교착 발생

**해결 과정** (3단계 진화):

```
Round 1: Standard A*
├─ 장점: CPU 효율 100%
├─ 문제: 교착 5~10%
└─ 결과: 불안정

Round 2: CBS-Lite (Conflict-Based Search)
├─ 장점: 교착 0%
├─ 문제: CPU 효율 82% (18% 감소)
└─ 결과: 느리지만 안정

Round 3: Priority A* 하이브리드 ✅ 채택
├─ 기본: Priority A* (혼잡도 기반 가중치)
│  └─ 간선 비용 = 기본값 × (1.0 + 혼잡도 × 2.5)
├─ 보조: CBS-Lite (8스텝 루킹어헤드)
│  └─ (NodeId, Timestep) 공간에서 미래 충돌 감지
└─ 결과: CPU 100% + 교착 0%
```

**성능 비교**:
| 알고리즘 | CPU 효율 | 교착 방지 | 처리율 |
|---------|---------|---------|--------|
| Standard A* | 100% | ❌ 5~10% | 22.5 items/s |
| CBS-Lite | 82% | ✅ 0% | 15.3 items/s |
| **Priority A* (채택)** | **100%** | **✅ 0%** | **19.0 items/s** |

### 2️⃣ 맵 설계: 단방향 그리드 (Unidirectional Grid Topology)

**원리**: 행/열 패리티를 이용한 일방향 이동

```
행 패리티:          열 패리티:
짝수 행 → 우측      짝수 열 → 하향
홀수 행 ← 좌측      홀수 열 ← 상향

결과: 2-cycle 구조적 불가능 → 정면충돌 0%
```

**설계 제약**:
- 특수 노드(D/E/Et/C 역, 창고)는 (짝수행, 짝수열) 부분격자에만 배치
- 이를 통해 특수 노드 간 비인접성 보장 + 균등 분포

**프리셋 (모두 단방향)**:
| 맵 | 크기 | 노드 수 | 공정역 | 용도 |
|----|------|--------|--------|------|
| 소형 | 6×4 | 24 | 4 | 학습용 |
| 중형 | 8×6 | 48 | 8 | 기본/테스트 |
| 대형 | 12×8 | 96 | 18 | **포트폴리오 데모** |
| 초대형 | 20×16 | 320 | 72 | 스트레스 테스트 |

### 3️⃣ 웹 ↔ Unity 포팅: TypeScript ↔ C# 동일성

**과제**: 같은 알고리즘을 두 언어/플랫폼에서 구현, 동일 결과 보장

**해결**:
1. **Web (TypeScript)**: 인터랙티브 데모 & 알고리즘 설명
2. **Unity (C#)**: 성능 최적화 & 3D 시각화
3. **검증**: 같은 맵 × 같은 시드 → 같은 경로 생성 ✅

**구현 예시** (Priority A*):
```typescript
// TypeScript (웹)
export function priorityAstar(from: NodeId, to: NodeId, congestion: Map<NodeId, number>): NodeId[]
```
```csharp
// C# (Unity)
public static List<int> PriorityAstar(MapNode from, MapNode to, Dictionary<string, float> congestion)
```

### 4️⃣ UX: 안정적인 경고 배너

**문제**: 경고 표시/숨김 시 레이아웃 밀림 (30px 점프) → "어지럽다"

**해결**:
- 고정 높이 (36px) 예약 + 투명도 전환 (opacity)
- 3초 메시지 유지 타이머
- 부드러운 fade-in/out 애니메이션

**결과**: 안정적인 UX + 가독성 개선

## 📚 부록: 기술 참고

### A. CBS-Lite 공간-시간 예약 알고리즘

```
시간 축:  t0  t1  t2  t3  t4  t5  t6  t7
         ┌─────────────────────────────┐ (8스텝 루킹어헤드)
Node A   │███│   │   │▒▒▒│   │   │   │
         └─────────────────────────────┘
                    ↑ 미래 충돌 감지
         (NodeId=A, Timestep=3) 예약됨 → ×8 페널티
```

### B. Priority A* 혼잡도 가중치

```
혼잡도 = (주변 8노드 중 점유된 노드 수) / 8

간선 비용 = base_cost × (1.0 + congestion × 2.5)

예시:
- 혼잡도 0 (빈 영역): 비용 = base_cost × 1.0
- 혼잡도 0.5 (중간): 비용 = base_cost × 2.25
- 혼잡도 1.0 (꽉 참): 비용 = base_cost × 3.5
```

### C. 단방향 그리드 토폴로지 증명

**명제**: 2-cycle (A→B→A)이 불가능하다.

**증명**:
1. A에서 B로 이동 가능 ⟹ A와 B의 행 또는 열이 다름
2. B에서 A로 이동하려면 A보다 역방향 이동 필요
3. 그런데 행/열 패리티에 의해 역방향 이동 불가 (같은 패리티는 진행방향 고정)
4. 따라서 B→A 이동 불가능 ✅
